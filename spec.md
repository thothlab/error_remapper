# Техническое задание: error-remapper

## 1. Назначение

Консольная утилита на Rust для поиска и подмены ошибок, приходящих от смежных бэкенд-систем. Утилита принимает JSON с ошибкой, ищет соответствие в YAML-словаре ошибок и возвращает переназначенный код и описание ошибки.

## 2. Входные данные

### 2.1. YAML-словарь ошибок (errors.yaml)

Формат:

```yaml
preprocess-error:
  vocabulary:
    - key: "2001"
      substring: "unexpected symbol:"
      customDesc: "Недопустимый символ %s в назначении перевода"
      code: "81002"
    - key: "2002"
      substring: "Уточните у получателя"
      code: "81001"
```

Поля каждой записи:
- `key` (обязательное) — код ошибки исходной системы
- `substring` (обязательное) — подстрока/паттерн для нечёткого поиска в тексте ошибки
- `code` (обязательное) — новый код ошибки (результат подмены)
- `customDesc` (необязательное) — кастомный текст ошибки для подмены

### 2.2. JSON ошибки от смежной системы (stdin или аргумент CLI)

Пример:
```json
{
  "error": {
    "code": -2,
    "title": "Сервер временно недоступен...",
    "uuid": "ddaa650d-...",
    "details": "..."
  }
}
```

Поля могут называться по-разному:
- Код ошибки: `code`, `errorCode`
- Текст ошибки: `title`, `message`, `errorMessage`

Список допустимых имён полей задаётся в файле настроек.

## 3. Алгоритм поиска

### 3.1. Шаг 1 — Точное совпадение по коду

1. Извлечь код ошибки из входного JSON (поле `code`/`errorCode`).
2. Найти все записи в YAML, где `key` совпадает с кодом ошибки (сравнение строковое, числовой код приводится к строке).
3. Если найдена **ровно одна** запись — перейти к формированию результата (шаг 3).

### 3.2. Шаг 2 — Нечёткий поиск по тексту

Выполняется если:
- По коду не найдено ни одной записи, ИЛИ
- По коду найдено несколько записей

Порядок:
1. **Подмножество записей** (если по коду найдено несколько): нечёткий поиск `substring` из каждой найденной записи по тексту `title` из входного JSON.
2. **Весь словарь** (если по коду записей не найдено): нечёткий поиск `substring` из всех записей по тексту `title`.
3. Выбрать запись с наивысшим скором совпадения.

### 3.3. Алгоритм нечёткого поиска

Применяется комбинированный подход:
1. **Точное вхождение подстроки** (case-insensitive) — приоритет, скор = 1.0
2. **Нормализованное расстояние Левенштейна** на скользящем окне — для случаев с опечатками и незначительными различиями.
3. **Порог отсечения** (настраиваемый, по умолчанию 0.4) — записи со скором ниже порога отбрасываются.

Библиотека: `strsim` (Rust crate) — предоставляет normalized Levenshtein, Jaro-Winkler и другие метрики.

### 3.4. Шаг 3 — Формирование результата

На основе найденной записи:
- `code` = значение поля `code` из YAML-записи
- `customDesc` = значение поля `customDesc` из YAML-записи, **если оно есть**; иначе — оригинальный текст ошибки (`title`) из входного JSON

## 4. Выходные данные

JSON в stdout:

```json
{"code": "81002", "customDesc": "Недопустимый символ в назначении перевода"}
```

При отсутствии совпадения — возврат оригинальных данных с флагом:

```json
{"code": "-2", "customDesc": "Сервер временно недоступен...", "matched": false}
```

## 5. Файл настроек (settings.toml)

```toml
[input]
code_fields = ["code", "errorCode"]
message_fields = ["title", "message", "errorMessage"]

[matching]
fuzzy_threshold = 0.4

[files]
errors_yaml = "errors.yaml"
```

## 6. CLI-интерфейс

```
error-remapper [OPTIONS] [INPUT_JSON]

Аргументы:
  [INPUT_JSON]          JSON-строка с ошибкой (если не указан — читает из stdin)

Опции:
  -c, --config <PATH>   Путь к settings.toml (по умолчанию: settings.toml)
  -e, --errors <PATH>   Путь к errors.yaml (перекрывает значение из settings.toml)
  -v, --verbose          Подробный вывод (логирование процесса поиска)
  -h, --help             Справка
  -V, --version          Версия
```
